## ADR-003 — DB 세팅 프로젝트와 서버 #3 DB 분리

- **날짜:** 2026-05-24
- **상태:** 확정

### 결정

- `JHSComputer_DataBase`를 별도 NestJS standalone 프로젝트로 둔다.
- SQL 파일은 `JHSComputer_DataBase/mysql/init` 아래에서 관리한다.
- 서버 #3(`data.hollywood.kro.kr`)의 한 MySQL 인스턴스 안에 `jhs_computer_dev`, `jhs_computer_stage`, `jhs_computer_prod`를 분리해서 둔다.
- `npm run db:bootstrap:server`는 세 DB와 각 DB 전용 계정을 만들고 schema/seed SQL을 적용한다.
- 로컬 개발과 초기 백엔드는 `jhs_computer_dev`만 사용한다.
- `npm run db:reset`은 dev DB만 대상으로 한다. `stage/prod`는 reset하지 않는다.
- SQL 파일이 DB 구조의 기준이고, 백엔드 TypeORM Entity는 이 SQL 기준을 따라간다.

### 이유

- 기존 C# Database 프로젝트처럼 실행 가능한 DB 세팅 단위를 만들기 위해서다.
- shell script만 두는 것보다 NestJS 실행기로 환경변수, 실행 순서, 로그를 일관되게 관리할 수 있다.
- dev/stage/prod를 DB 단위로 분리하면 로컬 개발, 배포 테스트, 운영 데이터를 섞지 않고 같은 서버에서 관리할 수 있다.
- 운영 migration 전략은 추후 별도로 정하고, MVP 단계에서는 schema/seed SQL 기준선을 먼저 안정화한다.

## ADR-004 — 부품 데이터와 공급처 상품 분리

- **날짜:** 2026-05-23
- **상태:** 확정

### 결정

- `parts`는 우리 서비스의 표준 부품이다.
- `supplier_products`는 컴퓨존/다나와 등 공급처별 상품 페이지다.
- `supplier_offers`는 공급처에서 실제 구매 가능한 판매 단위다.
- 같은 부품이라도 공급처 상품명, 상품번호, URL, 가격은 다르게 저장한다.
- 컴퓨존을 1차 공급처로 고려하되, DB는 다른 공급처를 추가할 수 있게 설계한다.

### 이유

- 공급처별 상품명이 다르다.
- 문자열 이름만으로 부품을 합치면 오매칭 위험이 크다.
- 향후 공급처가 늘어나도 견적/주문 구조를 유지해야 한다.

## ADR-005 — 가격 수집과 주문 직전 검증

- **날짜:** 2026-05-23
- **상태:** 확정

### 결정

- 사용자 화면은 우리 DB에 저장된 최신 가격을 보여준다.
- 백그라운드 가격 수집은 저빈도 작업으로 운영한다.
- 사용자가 주문하기를 누르면 견적에 포함된 상품만 즉시 가격 검증한다.
- 가격이 1원이라도 바뀌면 사용자 승인 상태로 전환한다.
- 확인 실패/차단 의심/재고 이상은 관리자 검토 상태로 전환한다.

### 이유

- 사용자 요청마다 전체 크롤링을 하면 차단 위험이 크다.
- 주문 직전 검증은 가격 분쟁을 줄인다.
- 전체 재수집은 큐에 넣고 천천히 처리해야 운영이 안정적이다.

### 주의

- 외부 사이트 수집은 약관/차단 정책/운영 리스크 검토가 필요하다.
- 서버 내부 수집 로직은 사용자 브라우저에 노출하지 않는다.

## ADR-006 — 견적은 Live, 주문은 Snapshot

- **날짜:** 2026-05-23
- **상태:** 확정

### 결정

- 견적은 파기하지 않고 최신 가격을 계속 반영한다.
- 주문은 주문 당시 부품명, 공급처 상품번호, URL, 가격, 배송정보를 스냅샷으로 저장한다.
- 견적 변경/가격 변경/호환성 결과는 이력으로 남긴다.

### 이유

- PC 부품 가격은 계속 변한다.
- 고객에게 최신 견적을 보여주면서도 주문 분쟁을 막으려면 스냅샷이 필요하다.
- 변경 이력은 고객 응대와 운영 감사에 필요하다.

## ADR-007 — 자동 견적은 템플릿 기반

- **날짜:** 2026-05-23
- **상태:** 확정

### 결정

- 가격대 + 용도 + 해상도 기준의 정효성 추천 템플릿을 만든다.
- 자동 견적은 템플릿을 기반으로 CPU/GPU 중심 조정을 수행한다.
- 게임은 템플릿이 아니라 성능 요구치로 매칭한다.

### 이유

- 완전 자유 조합은 호환성/품질/운영 리스크가 크다.
- 검증된 조합에서 출발하면 자동화 안정성이 높다.

## ADR-008 — 호환성 체크는 필수 조건

- **날짜:** 2026-05-23
- **상태:** 확정

### 결정

다음 항목을 모두 체크한다.

- CPU ↔ 메인보드 소켓
- 메인보드 ↔ RAM DDR 세대
- 케이스 ↔ 메인보드 규격
- 케이스 ↔ GPU 길이
- 케이스 ↔ CPU 쿨러 높이
- 파워 용량
- 저장장치 슬롯
- 쿨러 장착 가능 여부

### 이유

- 호환성은 사용자 취향이 아니라 주문 가능 여부의 필수 조건이다.
- 자동 견적 신뢰도는 호환성 품질에 의해 결정된다.

## ADR-009 — 커뮤니티는 후순위

- **날짜:** 2026-05-23
- **상태:** 확정

### 결정

- MVP에서는 커뮤니티를 만들지 않는다.
- 구매/견적/주문 기능 안정화 후 후기, 지식글, 게시판 순서로 확장한다.

### 이유

- 현재 가장 중요한 목표는 사이트 안에서 구매까지 완료되는 흐름이다.
- 커뮤니티를 먼저 만들면 핵심 구매 기능 개발 속도가 늦어진다.

## ADR-010 — 표준 부품과 판매 패키지 분리

- **날짜:** 2026-05-23
- **상태:** 확정

### 결정

- MVP에서는 별도 `part_model_groups`를 만들지 않고, `parts`를 견적에 들어가는 표준 부품 단위로 사용한다.
- 테이블명은 `TB_` 접두사 없이 `snake_case` 복수형을 사용한다.
- 컴퓨존 `ProductNo`, 다나와 `pcode`는 `supplier_products.external_product_id`에 저장한다.
- `external_product_id`는 공급처마다 의미가 달라도 `supplier_id + external_product_id`로 유니크하게 관리한다.
- CPU의 멀티팩, 정품박스, 벌크는 같은 CPU 실리콘과 동일 스펙이면 같은 `parts`에 연결한다.
- CPU 패키지 차이, 쿨러 포함 여부, 유통 형태, 가격 차이는 `supplier_offers`로 관리한다.
- RAM/SSD 용량, PSU 와트, 케이스/쿨러 색상처럼 견적 선택 조건이 달라지는 옵션은 서로 다른 `parts`로 본다.
- 다나와 스펙 표 원본은 수집 JSON 파일과 필요 시 `supplier_products.RAW_SPEC_JSON`에만 보관하고, DB에는 CPU/GPU/메인보드 등 카테고리별 정규화 스펙 테이블을 우선 저장한다.
- `raw_spec_sources`, `raw_spec_items`, `part_spec_values`, `quote_histories`, `price_check_logs`, `part_match_candidates`, `crawl_runs`, `crawl_items`, `compatibility_checks`는 MVP 테이블에서 제외한다.
- 주문 상태처럼 고객/운영에 의미 있는 이력만 별도 테이블로 남긴다.

### 이유

- 같은 `Ryzen 7 9800X3D`라도 컴퓨존과 다나와는 멀티팩/정품/해외구매를 서로 다른 판매 단위로 노출한다.
- 멀티팩과 정품박스는 보통 성능 스펙이 같고, 차이는 포장/구성품/쿨러 포함 여부/가격/유통 형태다.
- 판매 패키지와 기술 스펙을 섞으면 CPU 호환성 판단과 가격/마진 계산이 꼬인다.
- 별도 제품군 테이블을 두면 정교하지만 MVP DB가 과하게 복잡해진다.
- 호환성 체크는 정규화된 카테고리별 스펙을 기준으로 해야 안정적이다.
